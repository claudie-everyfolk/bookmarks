# Ashwin Gopinath: MCP moves context, it doesn't create it

**Author:** Ashwin Gopinath (@ashwingop)
**Date:** 2026-05-11
**URL:** https://x.com/ashwingop/status/2053874299757527109

## Tweet text
"MCP is the wire and memory is what flows through it. Connecting Claude to every tool feels magical, but tool access ≠ company context.

Real context exist between tools: decisions, owners, commitments, contradictions, permissions, state changes."

Companion article: "MCP Moves Context. It Doesn't Create It."

## Why it's relevant
Directly relevant to client engagements where the prospect demo'd a successful MCP integration but then can't extract real business value. The gap is exactly what Ashwin names: context that lives between tools (who decided what, what got committed, who owns it).

- **Natalia** — sharp diagnostic to use in discovery calls: "you've connected Claude to your tools — what context still lives outside them?" Identifies whether the prospect needs an MCP integration or a process-mapping engagement.
- Reinforces why discovery-to-process-map is a load-bearing skill — process maps capture the inter-tool context MCP can't.
- **Mike** — counter-narrative for the "just hook up MCPs and you're done" hype that students will hear elsewhere.
